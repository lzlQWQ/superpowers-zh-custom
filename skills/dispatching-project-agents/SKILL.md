# 分派工程子代理

按工程边界分派轻量子代理。开发计划涉及 `n` 个工程，就创建 `n` 个子代理；每个子代理只负责一个工程。主线程负责读取约束、设定规则、分配任务、跟踪状态，并按规范文档和开发计划审核产出。

**核心原则：** 一个工程一个子代理 + 主线程统一约束和审核 + 不跨工程修改。

## 触发边界

**只在用户显式要求时使用。** 普通多工程计划、普通“用子代理执行”、普通“执行计划”都不要自动触发本技能。

允许触发的用户表达：
- `dispatching-project-agents`
- “使用轻量子代理执行任务”
- “用轻量工程子代理执行”
- “按工程轻量分派子代理”
- “一个工程一个轻量子代理，由主线程审核”

不要触发本技能的表达：

| 用户表达 | 应使用 |
|----------|--------|
| “执行这个计划” | `superpowers:executing-plans` |
| “用子代理执行这个计划” | `superpowers:subagent-driven-development` |
| “用原子代理执行” | `superpowers:subagent-driven-development` |
| “用任务级子代理执行” | `superpowers:subagent-driven-development` |
| “subagent-driven-development” | `superpowers:subagent-driven-development` |
| “多个独立 bug 并行排查” | `superpowers:dispatching-parallel-agents` |

显式触发后，还必须确认计划确实适合按工程边界拆分：
- 同一计划同时修改前端、后端、管理端、移动端、服务端等多个工程
- 多个模块或服务有清晰目录边界和职责边界
- 主线程需要集中审查所有工程是否满足同一份规范和计划

## 流程

### 1. 读取约束

主线程必须先读取主约束：
- 优先读取当前工作区 `AGENTS.md`
- 如果 `AGENTS.md` 指向 `CLAUDE.md`，继续读取 `CLAUDE.md`
- 如果没有 `AGENTS.md`，读取 `CLAUDE.md`
- 解析主约束中要求读取的子约束、项目约束、目录约束或工程约束

**主约束优先级高于规范文档和开发计划。** 如果约束冲突，暂停并要求澄清。

### 2. 读取计划和规范

读取并整理：
- 规范文档
- 开发计划文档
- 主约束和子约束
- 每个工程的目录边界、允许修改路径、验证命令和依赖关系

如果开发计划没有明确工程边界，先从目录、模块、服务名和任务归属中整理工程清单；仍不明确时，先澄清再分派。

### 3. 建立分配表

创建并持续维护工程分配表：

| 工程 | 子代理 | 任务范围 | 允许修改路径 | 适用约束 | 依赖 | 状态 | 验证命令 | 审核结论 |
|------|--------|----------|--------------|----------|------|------|----------|----------|
| project-a | agent-a | ... | ... | ... | ... | PENDING | ... | ... |

状态只使用：
- `PENDING`
- `IN_PROGRESS`
- `DONE`
- `DONE_WITH_CONCERNS`
- `NEEDS_CONTEXT`
- `BLOCKED`
- `APPROVED`
- `REWORK_REQUIRED`

### 4. 分派子代理

每个工程分派一个子代理。使用 `project-agent-prompt.md` 模板，并填入：
- 工程名和任务范围
- 规范文档摘要
- 开发计划中该工程的完整任务
- 主约束摘要
- 必须读取的约束文件路径
- 工程专属约束
- 允许修改路径
- 禁止修改路径和共享文件规则
- 验证命令或验证策略

子代理不要读取整份开发计划来重新拆分任务。主线程必须给它完整、聚焦、可执行的工程任务。

### 5. 跟踪与协调

主线程负责：
- 跟踪每个子代理状态
- 统一回答问题
- 判断共享文件由哪个工程负责
- 阻止子代理跨工程扩散
- 记录阻塞、风险和未完成项

如果子代理返回 `NEEDS_CONTEXT`，补充上下文后重新分派给同一工程子代理。

如果子代理返回 `BLOCKED`，先判断是约束冲突、任务过大、依赖缺失还是工程边界错误；改变输入后再重试，不要原样重试。

### 6. 主线程审核

子代理返回后，主线程审核实际产出，而不是只信汇报。

审核必须覆盖：
- **约束符合性：** 是否遵守 `AGENTS.md` / `CLAUDE.md` 和子约束
- **计划符合性：** 是否完成开发计划中该工程的内容
- **规范符合性：** 是否满足规范文档要求
- **边界符合性：** 是否只修改允许路径，是否越权改共享文件
- **验证可信度：** 是否运行了约定验证，未运行时是否给出可验收路径

审核不通过时，将具体问题退回对应工程子代理修复，并把该工程状态标记为 `REWORK_REQUIRED`。

高风险工程或合并前需要更强质量把关时，再使用 `superpowers:requesting-code-review`。

## 红线

**绝不：**
- 在未读取主约束前分派子代理
- 让子代理自行猜测应遵守哪些约束
- 创建少于或多于工程数量的实现子代理
- 让一个子代理负责多个工程
- 允许多个子代理同时修改同一共享文件
- 让子代理跨工程修改
- 只根据子代理汇报宣布完成
- 在约束冲突未解决时继续实现

## 汇总格式

完成后向用户汇报：

```markdown
## 工程执行结果

| 工程 | 子代理状态 | 约束遵守 | 计划完成 | 验证结果 | 审核结论 |
|------|------------|----------|----------|----------|----------|
| ... | ... | ... | ... | ... | ... |

## 整体判断

- 已完成：
- 未完成：
- 风险：
- 需要用户验收：
```
