# 执行计划

加载计划，批判性审查，按任务执行，并按验证预算收集证据。

**开始时宣布：** "我正在使用 executing-plans 技能来实现此计划。"

**必需子技能：** 使用 `superpowers:testing-policy` 解释或补齐计划中的验证预算和测试策略。

如果平台支持子代理，优先使用 `superpowers:subagent-driven-development`；否则使用本技能内联执行。

## 步骤 1：加载并审查计划

1. 读取计划文件。
2. 提取任务、依赖、文件范围、验证预算、测试策略和验证检查点。
3. 检查计划缺陷：
   - 是否有任务缺少测试策略或验证命令？
   - 是否有任务缺少验证预算？
   - 是否把未定义方法/编译失败写成红灯？
   - 是否对多个小任务重复安排重型编译/测试？
   - 是否把高风险任务合并到过晚的检查点？
   - 是否存在不明确的“相关测试”“确认正常”？
4. 小缺口可按 `testing-policy` 补齐并记录；重大缺陷先向人类报告。

## 步骤 2：创建执行清单

为每个任务记录：

- 任务目标
- 测试策略：A/B/C/D
- 验证预算：是否允许重型验证、最多几次、已用几次
- 验证时机：立即 / 检查点 N / 完成前
- 验证命令、轻量核查方式或用户验收路径
- 是否需要 commit

## 步骤 3：执行任务

对每个任务：

1. 标记为进行中。
2. 重读任务目标和策略。
3. 实现计划中指定的变更。
4. 记录验证状态：
   - **A 强制 TDD：** 只在预算允许且红灯低成本时立即跑红灯和绿灯。
   - **B 实现优先目标验证：** 先完成实现，再并入目标验证或最终检查点。
   - **C 轻量核查：** 做静态阅读、diff 自审、关键调用链核对或 smoke check。
   - **D 交付用户验证：** 记录未运行项和用户验收路径。
5. 到达验证检查点时，运行合并命令并阅读输出。
6. 只有验证证据符合策略，才标记任务或检查点完成。

## 验证检查点

使用检查点减少重复测试和编译：

```text
检查点 1：任务 1-3
策略：B/C 合并验证
预算：最多 2 次重型验证；超时或超过 120 秒后停止重跑
命令：npm test tests/auth-flow.test.ts && npm run typecheck
完成条件：目标测试通过，类型检查 exit 0
```

检查点失败时：

1. 停止推进后续任务。
2. 根据输出定位最可能相关的任务。
3. 修复后重跑同一检查点。
4. 如果同一失败重复 2 次，或重型命令超时/超过 120 秒，停止重跑并报告状态。

## 常见异常

**无意义红灯：** 如果测试只因方法未定义、import 缺失或编译失败而失败，不得算作红灯。补最小结构后重跑行为测试。

**预算缺失：** 按 `testing-policy` 选择最低足够等级和重型验证次数，并在报告里写明理由。

**高风险任务被合并验证：** 改为即时验证，不等检查点。

**测试失败：** 区分实现 bug、测试基线、环境、命令问题；不要反复调测试消耗开发时间。

## 完成开发

所有任务和检查点完成后：

- 使用 `superpowers:verification-before-completion` 汇总证据。
- 使用 `superpowers:finishing-a-development-branch` 处理合并、PR 或保留分支。

## 完成报告

```markdown
## 执行报告

**计划：** docs/superpowers/plans/example.md
**任务：** 5/5 已完成

### 测试策略与验证
- 任务 1-2：C 轻量核查，并入检查点 1
- 任务 3：A 强制 TDD，红灯/绿灯均已运行
- 任务 4-5：B 实现优先目标验证，并入检查点 2

### 验证结果
- 验证预算：重型验证 1/2 次
- 检查点 1：`npm run typecheck` exit 0
- 检查点 2：`npm test tests/auth-flow.test.ts` 8/8 pass

### 偏离计划
- [如有，说明原因和验证影响]
```

## 注意事项

- 不要跳过验证；只允许按策略合并验证。
- 不要把编译失败当作红灯。
- 不要因为每个任务都很小就重复运行同一慢测试。
- 不要在重型命令超时后自动提高超时继续重跑。
- 未经用户明确同意，绝不在 main/master 分支上开始实现。
