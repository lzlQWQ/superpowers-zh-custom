# 测试策略

先选择测试策略，再写代码或执行计划。目标是质量不降级、验证次数下降。

## 核心规则

```
测试策略先于实现；完成结论必须有与风险匹配的新鲜证据。
```

不要把所有变更都套进红-绿循环。也不要用“节省时间”跳过必要验证。

## 策略分级

| 等级 | 适用场景 | 做法 |
|------|----------|------|
| **A. 必须 TDD** | bug 修复、行为回归、复杂业务逻辑、安全/支付/权限、公共 API 行为变更 | 先写能进入断言层的失败测试，确认因预期行为失败，再实现并跑通过 |
| **B. 目标测试** | 新增可测试行为，但红灯会被未定义方法、缺失类型或脚手架阻断 | 先补最小结构让测试能编译，再写真实行为测试；可先实现后补目标行为测试 |
| **C. 轻量验证** | 简单透传、方法签名、类型适配、配置、样式、文案、生成代码、小范围重构 | 跑构建、类型检查、lint、相关 smoke check 或人工可复现检查 |
| **D. 可跳过自动化测试** | 纯文档、注释、README、非执行性元数据，或用户明确要求不跑测试 | 说明无需自动化测试的理由，必要时做格式/链接/渲染检查 |

选择最低且足够证明风险的等级。风险升高时升级策略。

## 无意义红灯

以下失败不得算作有效红灯：

- 方法、类、字段、模块尚未定义导致编译失败
- import、类型声明、测试脚手架缺失
- 测试框架配置错误，测试没有进入断言
- Maven/TypeScript/编译器在业务断言前失败

遇到这种情况：

1. 补最小结构，让测试能编译并执行到断言。
2. 对真实业务行为写测试。
3. 只有当失败原因是预期行为缺失，才算红灯。

## 合并验证

小任务可以共享验证检查点。满足以下条件时合并：

- 任务相邻且改动风险低到中等
- 目标测试或构建命令相同
- 任一任务失败时仍能定位责任范围

不要合并：

- 高风险任务
- 会影响大量模块的共享接口
- 已经出现失败或不确定性的任务

## 汇报格式

在计划、执行或完成报告中写清楚：

- **测试策略：** A/B/C/D
- **理由：** 为什么这个等级足够
- **验证检查点：** 立即验证或并入哪个检查点
- **验证证据：** 已运行命令、人工检查或跳过理由

## 与其他技能的关系

- `test-driven-development`：仅在策略 A，或策略 B 进入真实行为测试后使用。
- `writing-plans`：每个任务先写测试策略，再写实现步骤和验证检查点。
- `executing-plans` / `subagent-driven-development`：记录每个任务的策略，按检查点批量验证。
- `verification-before-completion`：完成前仍必须验证，但验证类型由本策略决定。
