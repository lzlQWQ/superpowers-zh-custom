# 测试策略与验证预算

先选择验证预算，再写代码或执行计划。目标是快速、正确地完成代码改动，把重型编译和测试作为受控资源使用。

## 核心规则

```
验证预算先于执行；重型编译/测试受预算控制；未验证的结论必须如实表述。
```

默认前提：用户会自行做最终业务验收。Agent 的职责是高效完成改动、做必要轻量核查，并给出清晰的用户验收路径。不要把所有变更都套进红-绿循环，也不要把慢编译当作每个小步骤的默认动作。

## 重型验证

以下都算重型验证：

- Maven/Gradle 编译或测试
- 全量测试、模块级测试、集成测试、E2E
- 前端完整 build
- 历史上或预期会超过 30 秒的命令
- 需要大量 mock、stub、环境准备或外部服务的测试

默认预算：

- 业务代码修改前：重型验证次数为 0，除非策略 A 明确要求。
- 普通任务完成前：最多 2 次重型验证。
- 多个小任务：共享一个最终检查点，不要每个任务各跑一次。
- 命令超时或超过 5 分钟：停止自动重跑，报告状态和建议用户执行的验证命令。
- 同一重型命令在没有业务代码变化前不得重复运行。

## 策略分级

| 等级 | 适用场景 | 做法 |
|------|----------|------|
| **A. 强制 TDD** | 安全、支付、权限、数据一致性、公共协议/API，或用户明确要求回归测试 | 只有红灯能低成本进入断言层时才先写失败测试；否则降级为 B |
| **B. 实现优先目标验证** | 大多数业务 bug 修复、局部行为修正、已有根因证据的改动 | 先实现最小业务修复，再选择轻量核查或 1 次目标验证 |
| **C. 轻量核查** | 简单透传、方法签名、类型适配、配置、样式、文案、小范围重构 | 静态阅读、diff 自审、关键调用链核对、局部 smoke check |
| **D. 交付用户验证** | 重型验证不划算、环境成本高、用户明确会验收、纯文档/注释/元数据 | 不跑重型验证；输出改动说明和人工验收路径 |

选择最低且足够控制风险的等级。开发速度是默认优化目标；风险升高时再升级策略。

## TDD 准入

只有同时满足以下条件，才允许策略 A：

- 红灯能进入断言层，而不是停在编译、import、脚手架或环境错误。
- 测试运行成本低，不依赖重型编译或长时间执行。
- 测试 setup 不需要复杂 mock/stub。
- 红灯能提供新增事实，而不是重复已经定位的根因。

如果不满足任一条件，使用策略 B：先实现，再做目标验证或交付用户验证。

以下失败不得算作有效红灯：

- 方法、类、字段、模块尚未定义导致编译失败
- import、类型声明、测试脚手架缺失
- 测试框架配置错误，测试没有进入断言
- Maven/TypeScript/编译器在业务断言前失败

遇到无效红灯，不要继续修测试来制造红灯。改用策略 B 或 C。

## 合并验证

小任务默认共享验证检查点。满足以下条件时必须合并：

- 任务相邻且改动风险低到中等
- 目标测试或构建命令相同
- 命令属于重型验证
- 任一任务失败时仍能定位责任范围

不要合并：

- 策略 A 的高风险任务
- 会影响大量模块的共享接口
- 已经出现失败或不确定性的任务

## 失败处理

- 重型验证失败时，先判断是实现问题、测试基线问题、环境问题还是命令问题。
- 如果失败来自 mock/stub/断言文案/测试基线，最多修一轮。
- 修一轮后仍失败，停止测试调试，报告真实状态和用户验收建议。
- 不要通过不断提高超时、扩大测试范围或反复调整测试来消耗开发时间。

## 汇报格式

在计划、执行或完成报告中写清楚：

- **测试策略：** A/B/C/D
- **理由：** 为什么这个等级足够
- **验证预算：** 是否允许重型验证、最多几次、已用几次
- **验证检查点：** 立即验证、并入哪个检查点，或交由用户验收
- **验证证据：** 已运行命令、轻量核查、人工检查或跳过理由
- **用户验收路径：** 未运行重型验证时必须提供

## 与其他技能的关系

- `test-driven-development`：仅在策略 A 使用。
- `writing-plans`：每个任务先写验证预算、测试策略、实现步骤和检查点。
- `executing-plans` / `subagent-driven-development`：记录预算使用，按检查点批量验证，避免重复重型命令。
- `verification-before-completion`：区分“代码改动完成”和“测试/编译通过”；验证类型和预算由本策略决定。
