# 编写计划

编写可交给另一个工程师或 agent 执行的实现计划。计划要先锁定纵向任务拆分方案，获得用户批准后，再把每个任务写成独立计划文件。

**开始时宣布：** "我正在使用 writing-plans 技能创建实现计划。"

**必需子技能：** 先使用 `superpowers:testing-policy` 为每个任务选择验证预算和验证等级。

**计划保存位置：** `docs/superpowers/YYYY-MM-DD-<feature-name>/plans/NN-<slice-name>.md`
用户指定其他根目录时可以调整根目录，但必须保留 `YYYY-MM-DD-<feature-name>/plans/NN-<slice-name>.md` 结构。

## 计划原则

- 每个任务交付一条窄而完整、可独立验收的价值路径：从触发条件到可观察结果，覆盖必要实现与验证，不交付只完成内部某层或某工序的半成品。
- 每个任务必须单独保存为一个计划文件，文件名不带日期，使用 `01-<slice-name>.md`、`02-<slice-name>.md` 排序。
- 日期只放在任务目录名前面：`YYYY-MM-DD-<feature-name>`。
- 任务粒度按用户可感知结果、业务能力、操作闭环或可验证行为划分；不要按技术层、工序阶段、文件类型、职责分层或验证活动横向划分。
- 如果一条价值路径仍然过大，继续拆成更薄、仍可独立验收的路径。
- 验证成本不得随任务数量线性增加；能后置的验证放到后方，能合并执行的验证合并到检查点。
- 只有高风险、不可延后或会阻塞后续判断的验证才即时执行，但仍要写明重型验证预算。
- 不要求每个任务都 TDD；由 `testing-policy` 判断 A/B/C/D。
- 每个计划必须说明哪些验证立即执行，哪些合并到检查点执行，哪些交由用户验收。
- 不创建总计划 `index.md` 或单一总计划文件，避免重新形成“一个文档内多个任务”的结构。

## 拆分确认关卡

写入任何计划文件之前，必须先在对话框展示拆分方案并询问用户是否可以生成执行计划。

拆分方案使用编号列表，每个任务显示：

- **Title：** 短任务名
- **Goal：** 这个任务交付的可验收能力
- **Blocked by：** 依赖哪些任务；没有则写 None
- **需求覆盖：** 覆盖哪些明确需求、约束或验收点
- **Validation：** 该任务完成后如何独立 demo 或验证

询问用户：

> 以上拆分是否可以生成执行计划？如果粒度太粗、太细、依赖关系不对，或需要合并/继续拆分，请告诉我。

只有用户批准后，才能创建 `docs/superpowers/YYYY-MM-DD-<feature-name>/plans/` 并写入计划文件。

## 目录结构

```text
docs/superpowers/YYYY-MM-DD-<feature-name>/
  specs/<feature-name>-design.md
  plans/01-<slice-name>.md
  plans/02-<slice-name>.md
```

`<feature-name>` 和 `<slice-name>` 默认使用用户确认名称的 kebab-case。

## 每个计划文件的头部

每个计划文件只描述一条独立价值路径：

```markdown
# [任务名称] 执行计划

> **面向 AI 代理的工作者：** 必需子技能：使用 superpowers:testing-policy 选择验证预算和测试策略；使用 superpowers:subagent-driven-development（推荐）或 superpowers:executing-plans 执行。步骤使用复选框（`- [ ]`）语法跟踪进度。

**所属任务：** [YYYY-MM-DD-<feature-name>]
**计划文件：** [plans/NN-<slice-name>.md]
**Blocked by：** [None / plans/01-xxx.md]

**目标：** [一句话描述这个任务交付什么可验收能力]
**端到端范围：** [触发入口、关键处理、状态变化、外部影响和验证证据；不适用的环节写“无”]

**技术栈：** [关键技术/库]

**验收标准：**
- [ ] [可验证结果 1]
- [ ] [可验证结果 2]

**测试策略：** A 强制 TDD / B 实现优先目标验证 / C 轻量核查 / D 交付用户验证
**策略理由：** [为什么这个等级足够；如无需红灯，写清楚原因]
**验证预算：** [是否允许重型验证；最多几次；已知慢命令如何处理]
**验证检查点：** 立即执行 / 并入检查点 M / 完成前统一执行
**用户验收路径：** [触发方式、输入条件、可观察结果；不适用则写“无”]

---
```

## 任务模板

````markdown
## 实施步骤

**文件：**
- 创建/修改：`exact/path/to/file`
- 测试：`tests/exact/path/to/test`（如适用）

- [ ] 实现 [具体行为或结构变更]

[必要时给出关键代码片段、接口签名或伪代码。不要写占位符。]

- [ ] 按策略记录验证证据和预算使用

运行：`exact command`
预期：`具体结果`，或说明轻量核查/用户验收方式。
````

计划文件中不要再写“任务 1 / 任务 2 / 任务 3”的横向任务列表。一个文件就是一个任务。

## TDD 任务写法

只有 `testing-policy` 判定为 A 时，才在任务内写红-绿步骤：

```markdown
**测试策略：** A 强制 TDD
**策略理由：** 修复支付状态回归，必须先证明测试能捕获原始 bug。
**Blocked by：** None
**验证预算：** 允许 1 次红灯和 1 次绿灯目标测试；不运行完整套件。
**验证检查点：** 立即执行

- [ ] 写回归测试，测试必须执行到断言并因原始症状失败
- [ ] 运行 `npm test tests/payment-status.test.ts -t "keeps paid order closed"`，预期 FAIL，原因是订单状态被错误重开
- [ ] 实现最小修复
- [ ] 重新运行同一测试，预期 PASS
```

如果红灯需要重型编译、复杂 mock/stub、长时间运行，或只会得到“方法未定义”/编译失败，不要写成 TDD 任务；使用策略 B，先实现再做目标验证或用户验收。

## 禁止占位符

不要写：
- "待定"、"TODO"、"后续实现"
- "添加适当的错误处理"
- "为上述代码编写测试"
- "运行相关测试"
- "确认功能正常"
- "类似任务 N"

必须写具体文件、命令、预期结果和策略理由。

## 自检

计划完成后检查：

1. **需求覆盖：** 每个需求都能映射到任务。
2. **策略覆盖：** 每个任务都有测试策略、理由、验证预算和验证检查点。
3. **无意义红灯：** 没有把未定义方法、缺失 import、编译失败写成有效红灯。
4. **验证预算：** 重型编译/测试没有按小任务重复运行。
5. **执行完整：** 没有占位符、隐含依赖或不明确命令。
6. **纵向拆分：** 每个任务都按用户可感知结果、业务能力、操作闭环或可验证行为拆分，而不是按内部实现分层、工序阶段、文件类型或验证活动拆分。
7. **依赖声明：** 每个计划文件都有 `Blocked by`，且依赖关系可执行。

## 执行交接

保存计划后，提供执行选项：

**"计划已完成并保存到 `docs/superpowers/YYYY-MM-DD-<feature-name>/plans/`。两种执行方式：**

**1. 子代理驱动（推荐）** - 按 `NN-*.md` 顺序读取计划，校验 `Blocked by` 后按依赖、相关性和任务量调度子代理，并进行规格/质量审查

**2. 内联执行** - 在当前会话中使用 executing-plans 按目录执行计划文件，按验证检查点推进

**选哪种方式？"**
