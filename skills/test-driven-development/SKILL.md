# 测试驱动开发（TDD）

先用 `testing-policy` 选择策略。只有策略 A（必须 TDD）或策略 B 中真实行为测试阶段，才执行本技能。

## 适用边界

**必须使用 TDD：**
- bug 修复和回归测试
- 复杂业务规则、边界条件、状态机
- 安全、权限、支付、数据完整性
- 对外 API、公共方法或协议的行为变更
- 任何“测试先失败才能证明修复有效”的场景

**不要强行使用 TDD：**
- 新增方法签名、空实现、DTO、类型适配
- 简单透传、配置、样式、文案、文档
- 生成代码或机械性迁移
- 红灯只能得到编译失败，无法进入断言层的场景

这些场景按 `testing-policy` 使用目标测试、构建/类型检查或跳过自动化测试说明。

## 铁律

```
有效红灯 = 测试能执行到断言，并因预期行为缺失而失败。
```

无效红灯不算 TDD。尤其是：
- 方法、类、字段、模块尚未定义
- import 或类型声明缺失
- Maven/TypeScript/编译器在测试断言前失败
- 测试框架或环境配置错误

遇到无效红灯时，先补最小结构让测试能编译并进入断言，再写真实行为测试。

## 红-绿-重构

### 1. 红：写失败的行为测试

写一个最小测试，只表达一个期望行为。

```typescript
test('retries failed operations 3 times', async () => {
  let attempts = 0;
  const operation = () => {
    attempts++;
    if (attempts < 3) throw new Error('fail');
    return 'success';
  };

  const result = await retryOperation(operation);

  expect(result).toBe('success');
  expect(attempts).toBe(3);
});
```

要求：
- 名称描述业务行为
- 只测一件事
- 优先使用真实代码，mock 只用于不可控边界

### 2. 验证红灯

运行最小目标测试命令。

```bash
npm test path/to/test.test.ts
```

确认：
- 测试执行到断言
- 失败原因是预期行为缺失
- 不是编译、导入、测试脚手架或环境错误

如果失败原因无效，先补最小结构或修测试环境，再重新验证红灯。

### 3. 绿：写最少实现

只写让测试通过的代码。不要顺手加未要求的功能、重构或“以后可能用到”的选项。

### 4. 验证绿灯

重新运行同一个目标测试。通过后，根据 `testing-policy` 决定是否需要相关测试、构建、类型检查或更大范围回归。

### 5. 重构

只有在绿灯后重构。重构后重新运行足以覆盖该行为的验证命令。

## Bug 修复模式

1. 先重现原始 bug。
2. 写能失败的回归测试，且失败原因必须是原始症状。
3. 实现最小修复。
4. 跑回归测试通过。
5. 跑与风险匹配的相关验证。

不要在没有回归测试或复现证据的情况下宣布 bug 已修复。

## 完成前报告

汇报时包含：
- **测试策略：** A 或 B
- **红灯证据：** 命令和预期失败原因
- **绿灯证据：** 命令和通过结果
- **后续验证：** 相关测试、构建、类型检查或合并检查点

如果没有有效红灯，不要声称完成了 TDD；说明采用了 `testing-policy` 的其他策略。
