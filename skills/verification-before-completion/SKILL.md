# 完成前验证

在没有验证的情况下宣称工作完成，是不诚实。验证不一定总是完整测试套件，但必须与风险匹配。

**必需子技能：** 使用 `superpowers:testing-policy` 判断需要哪类证据。

## 铁律

```
没有新鲜的、与风险匹配的验证证据，不许宣称完成。
```

如果你在这条消息中没有运行或核对验证，就不能声称“测试通过”“构建通过”“已修复”或“完成”。

## 门控函数

在宣称任何成功状态之前：

1. **确定结论：** 你要证明什么？完成、修复、构建通过、测试通过，还是需求满足？
2. **选择证据：** 按 `testing-policy` 选择验证等级：
   - A：TDD 红灯/绿灯和相关回归
   - B：目标行为测试
   - C：构建、类型检查、lint、smoke check 或人工可复现检查
   - D：跳过自动化测试的明确理由和必要文档/格式检查
3. **运行或核对：** 执行完整命令，或完成计划指定的人工检查。
4. **阅读输出：** 检查退出码、失败数、错误和警告。
5. **对齐结论：** 输出支持结论才可声明；否则报告真实状态。

跳过任何一步都不是验证。

## 合格证据

| 结论 | 合格证据 | 不够格 |
|------|----------|--------|
| 测试通过 | 本轮目标测试或检查点输出：0 failures | 之前的运行结果、推断 |
| 构建通过 | 构建命令 exit 0 | lint 通过但未构建 |
| 类型正确 | 类型检查 exit 0 | 代码看起来类型没问题 |
| Bug 已修复 | 原始症状复现测试或人工复现路径通过 | 只改了代码 |
| 回归测试有效 | 有效红灯进入断言层，再绿灯通过 | 只看到编译失败 |
| 需求已满足 | 计划核对清单 + 对应验证证据 | 单个测试通过 |
| 无需自动化测试 | 策略 D 理由 + 文档/格式/人工检查结果 | “太小了不用测” |

## 红线

- 使用“应该”“大概”“看起来”
- 验证前表达完成或满意
- 把未定义方法、import 缺失、编译失败当作红灯
- 因为测试慢就完全不验证
- 信任子代理的成功报告而不检查证据
- 依赖部分输出但声称完整通过

## 验证报告

完成报告必须包含：

```markdown
**测试策略：** A/B/C/D
**验证命令或检查：** `exact command` / 人工检查步骤
**结果：** exit code、通过数量、失败数量或检查结果
**结论：** 输出支持什么，不支持什么
```

如果没有运行某类测试，直接说明：“未运行完整测试套件；本次按策略 C 运行 `npm run typecheck`，因为变更仅涉及类型适配。”

## 代理委派

子代理报告成功后：

1. 检查它报告的测试策略。
2. 检查实际 diff。
3. 核对验证命令和输出是否支持结论。
4. 必要时由控制者重新运行检查点。

不要把代理报告当作证据本身。

## 底线

验证可以分级，不能消失。用合适的证据说合适的话。
