# 完成前验证

汇报必须精确。可以说“代码改动已完成”或“已做静态核查”，但只有实际运行成功后才能说“测试通过”“编译通过”或“已验证修复”。

**必需子技能：** 使用 `superpowers:testing-policy` 判断需要哪类证据。

## 铁律

```
没有对应证据，不许声明对应结论。
```

如果未运行重型验证，可以交付代码改动，但必须明确：“未运行编译/测试；建议用户按以下路径验收”。

## 门控函数

在宣称任何成功状态之前：

1. **确定结论：** 你要证明什么？完成、修复、构建通过、测试通过，还是需求满足？
2. **选择证据：** 按 `testing-policy` 选择验证预算和策略：
   - A：强制 TDD，且红绿灯在预算内
   - B：实现优先目标验证，可用一次目标检查或交付用户验收
   - C：轻量核查，如静态阅读、diff 自审、关键调用链核对、smoke check
   - D：交付用户验证，说明未跑重型验证和人工验收路径
3. **运行或核对：** 执行预算内命令，或完成计划指定的轻量核查。
4. **阅读输出：** 检查退出码、失败数、错误和警告。
5. **对齐结论：** 输出支持结论才可声明；否则报告真实状态。

不要为了获得“测试通过”结论而突破重型验证预算。

## 合格证据

| 结论 | 合格证据 | 不够格 |
|------|----------|--------|
| 测试通过 | 本轮目标测试或检查点输出：0 failures | 之前的运行结果、推断 |
| 构建通过 | 构建命令 exit 0 | lint 通过但未构建 |
| 类型正确 | 类型检查 exit 0 | 代码看起来类型没问题 |
| 代码改动已完成 | diff 自审、关键路径核对、文件改动说明 | 声称测试或编译通过 |
| 交付用户验收 | 未运行项说明 + 可执行验收路径 | 只说“请测试” |
| Bug 已验证修复 | 原始症状复现测试或人工复现路径通过 | 只改了代码 |
| 回归测试有效 | 有效红灯进入断言层，再绿灯通过 | 只看到编译失败 |
| 需求已满足 | 计划核对清单 + 对应验证证据 | 单个测试通过 |
| 未跑重型验证 | 策略 B/C/D 理由 + 用户验收路径 | 隐瞒未验证范围 |

## 红线

- 使用“应该”“大概”“看起来”
- 验证前表达完成或满意
- 把未定义方法、import 缺失、编译失败当作红灯
- 因为测试慢就隐瞒未验证范围
- 超出 `testing-policy` 的重型验证预算反复重跑
- 信任子代理的成功报告而不检查证据
- 依赖部分输出但声称完整通过

## 验证报告

完成报告必须包含：

```markdown
**测试策略：** A/B/C/D
**验证预算：** 是否使用重型验证、已用次数、是否超时/跳过
**验证命令或检查：** `exact command` / 人工检查步骤
**结果：** exit code、通过数量、失败数量或检查结果
**结论：** 输出支持什么，不支持什么
**用户验收路径：** 未运行重型验证时必填
```

如果没有运行编译或测试，直接说明：“未运行编译/测试；建议用户按以下路径验收：...”。

## 代理委派

子代理报告成功后：

1. 检查它报告的测试策略。
2. 检查实际 diff。
3. 核对验证命令和输出是否支持结论。
4. 必要时由控制者重新运行检查点。

不要把代理报告当作证据本身。

## 底线

验证可以是轻量核查或用户验收，不能伪装成已通过的自动化证据。用合适的证据说合适的话。
